# Функциональное ревью и план сужения Narrative Intelligence Platform

Документ фиксирует новую продуктовую рамку MVP. Цель — перестать распыляться на все возможные intelligence-фичи и довести до сильного уровня четыре сценария:

1. Анализ статьи.
2. Подбор похожих статей.
3. Граф связей.
4. Сравнение освещения.

Все остальное стоит либо убрать из главной навигации, либо оставить как внутренний вспомогательный слой.

## 1. Сужение функционала

### Оставить как ядро

**Анализ статьи**

- Пользователь выбирает статью и получает понятный анализ на русском: краткое резюме, что произошло, кто участники, тональность, framing, кому текст симпатизирует, кого критикует, гипотеза нарратива.
- Главная ценность — не просто JSON от LLM, а объяснимость: цитаты/evidence должны показывать, почему система сделала вывод.
- Кнопка `Анализ` должна явно показывать состояние: `ожидает`, `анализируется`, `готово`, `ошибка Ollama`, `нужно запустить Ollama`.

**Похожие статьи**

- Это должен быть не просто nearest-neighbor из Qdrant, а гибридный подбор: embedding similarity + пересечение сущностей + пересечение ключевых слов + близость по времени.
- Если похожих статей нет, лучше честно показать пустое состояние, чем соединять ФАС с Британией и санкциями из-за общего экономического контекста.
- Для каждой похожей статьи нужно показывать причину: общие сущности, общий факт, общий event, общий narrative frame.

**Графы**

- Граф должен быть главным способом навигации, но не “все связи сразу”.
- Базовый режим: статья в центре, рядом только источник, ключевые сущности, нарративная гипотеза и реально похожие статьи.
- Расширенный режим: пользователь сам включает дополнительные слои: evidence, weak links, entities, related articles.
- Граф обязан быть стабильным: при zoom/pan он не должен прыгать, перестраиваться или менять масштаб без действия пользователя.

**Сравнение**

- Сравнение должно запускаться только для статей, которые прошли фильтр похожести.
- Основной результат: совпадающие факты, различия, framing каждой стороны, симпатии/критика, narrative difference, вывод.
- Нужно добавить “качество сравнения”: вероятность, что статьи об одном событии, и предупреждение, если это разные события.

### Убрать из первого слоя MVP

**Events**

- Event layer полезен, но сейчас он должен работать как backend-support для похожих и графов.
- Отдельную вкладку Events можно скрыть из основного UI до тех пор, пока качество event matching не станет стабильным.

**Source Profile**

- Хорошая будущая аналитика, но сейчас она отвлекает.
- Можно оставить endpoint, но убрать из главной навигации.

**Reports / Import / broad command center**

- Не делать вид, что это готовая enterprise-платформа.
- Для MVP важнее: загрузил статьи, проанализировал, увидел похожие, открыл граф, сравнил.

## 2. Как улучшить качество ключевых функций

### Анализ

Проблема: LLM может возвращать слишком уверенные, но слабодоказанные выводы.

Что улучшить:

- Разделить анализ на два шага:
  - `fact extraction`: факты, участники, даты, места, цитаты.
  - `interpretation`: framing, stance, sympathy, criticism, narrative hypothesis.
- Валидировать ответ через Pydantic-схемы, а не только через ручной JSON parsing.
- Сохранять raw LLM response для отладки.
- Добавить evidence coverage score: сколько ключевых выводов подтверждены цитатами.
- Ограничить длину текста для Ollama умнее: title + lead + первые N абзацев + последние N абзацев + уже найденные сущности.
- Для английских статей сохранять и показывать русскую версию summary/framing/narrative, но не терять оригинальный title/url.

### Похожие статьи

Проблема: чистый embedding ловит общий тематический фон, но не гарантирует, что статьи реально связаны.

Текущий правильный вектор развития:

- Qdrant дает кандидатов.
- Потом backend применяет guard:
  - общие важные сущности;
  - общие ключевые слова;
  - близкая дата;
  - не является техническим дублем;
  - не слишком общий overlap вроде “Россия”, “США”, “санкции”.
- Для top-кандидатов можно добавить быстрый LLM rerank:
  - “Это тот же сюжет, связанный сюжет или просто общая тема?”
  - Возвращать `same_story`, `related_context`, `not_related`.
- В UI показывать не только score, а объяснение: “общие сущности: ФАС, нефтяные компании; общая тема: цены на топливо”.

### Графы

Проблема: граф визуально уже близко, но управляемость нестабильна: скачки, дергание, странный zoom.

Что улучшить не косметически, а архитектурно:

- Разделить frontend-граф на модули:
  - `graphScene`: Three.js lifecycle, renderer, cleanup.
  - `graphLayout`: координаты узлов, режимы, стабильное центрирование.
  - `graphEncoding`: размер, цвет, opacity, confidence.
  - `graphInteraction`: hover, click, selection, fly-to, zoom.
  - `GraphDetailsPanel`: правая панель.
- Сделать layout deterministic: один и тот же graph JSON всегда дает одни и те же координаты.
- Не пересоздавать всю Three.js-сцену на каждое изменение hover/selection.
- Отключить auto-orbit по умолчанию.
- Добавить мягкий camera clamp:
  - min/max distance;
  - ограничение pan;
  - reset к bounding box графа;
  - fly-to без изменения scale сцены.
- Ввести два режима управления:
  - `Explore`: свободный pan/zoom/rotate.
  - `Focus`: камера фиксируется на выбранном узле и показывает только соседей.
- Для подписей:
  - всегда показывать статью, источник и top-нарратив;
  - остальные подписи показывать по label density;
  - не позволять labels перекрывать друг друга;
  - по hover показывать label всегда.

### Сравнение

Проблема: сравнение отлично выглядит, но зависит от качества похожих статей.

Что улучшить:

- Перед сравнением всегда запускать pre-check:
  - same event probability;
  - fact overlap;
  - entity overlap;
  - date distance.
- Если статьи разные, показывать это как результат, а не притягивать сравнение.
- Добавить табличный блок:
  - `Факт`;
  - `Источник 1`;
  - `Источник 2`;
  - `Совпадает / отличается / отсутствует`.
- Добавить блок “Главное расхождение” поверх карточек.
- Для каждой разницы показывать evidence quote из обеих статей.

## 3. Дизайн и UX

### Общий стиль

Сейчас проект иногда выглядит как дорогой dashboard, а иногда как набор разных экранов. Нужно привести все к одной дизайн-системе.

Решение:

- Оставить dark intelligence style, но сделать его спокойнее.
- Уменьшить количество glow, gradients и dense cards.
- Использовать один набор компонентов:
  - `PageShell`;
  - `TopBar`;
  - `FilterBar`;
  - `MetricCard`;
  - `ArticleCard`;
  - `InsightPanel`;
  - `ActionButton`;
  - `StatusBadge`.
- На главной странице оставить меньше визуального шума:
  - карточки статей компактнее;
  - меньше nested panels;
  - right panel показывать только полезные инсайты;
  - если данных нет, не имитировать аналитику.

### Что поправить в графах

Графы визуально можно оставить в текущем направлении, но управление нужно чинить как продуктовую проблему.

Главные задачи:

- Стабилизировать camera controls.
- Зафиксировать начальный центр по bounding box графа.
- Исключить полную пересборку сцены при мелких UI-действиях.
- Сделать понятные режимы:
  - `Статья`;
  - `Похожие`;
  - `Сущность`;
  - `Сравнение`.
- Убрать режимы, которые не дают понятной ценности.
- Добавить “breadcrumb движения” по графу: от какой статьи пришли, куда перешли, почему связь существует.

## 4. Backend-подготовка данных, чтобы пользователь не ждал

Сейчас часть действий все еще ощущается синхронной: пользователь нажимает `Анализ`, `Похожие`, `Граф`, а система может начать тяжелую работу прямо в запросе.

Правильная модель:

### Pipeline после загрузки статей

После ingestion должен запускаться backend pipeline:

1. Дедупликация и canonical URL.
2. LLM-анализ.
3. Evidence validation.
4. Entity/relation extraction.
5. Embedding.
6. Similar candidates.
7. Similar guard/rerank.
8. Graph JSON cache.
9. Optional compare cache для top-3 похожих.

### Job layer

Нужно добавить таблицу `jobs`:

- `id`;
- `type`;
- `status`;
- `progress`;
- `started_at`;
- `finished_at`;
- `params_json`;
- `result_json`;
- `error`;
- `retry_count`.

Пока можно без Celery: локальный worker внутри FastAPI или отдельная команда `python -m app.worker`.

### Cache policy

Для быстрых UI-действий хранить:

- `article_analysis`;
- embedding в Qdrant;
- `similar_json`;
- `graph_json`;
- `compare_json`;
- версии алгоритмов:
  - `analysis_prompt_version`;
  - `embedding_model`;
  - `similarity_guard_version`;
  - `graph_schema_version`.

Если версия изменилась, cache не используется.

### UI statuses

На карточке статьи нужен один понятный readiness badge:

- `raw`: статья загружена, анализа нет;
- `analyzed`: есть LLM-анализ;
- `embedded`: можно искать похожие;
- `ready`: похожие и граф подготовлены;
- `failed`: есть ошибка обработки.

Тогда пользователь понимает, почему кнопка доступна или недоступна.

## 5. Вкладка Narratives

Текущее состояние: вкладка выглядит слабее остального продукта и функционально пока не доказывает ценность.

Решение для MVP:

- Не делать Narratives отдельной центральной вкладкой.
- Перенести narrative hypothesis внутрь:
  - карточки анализа статьи;
  - graph details;
  - compare page;
  - right insight panel.
- Оставить narrative discovery как экспериментальный backend-инструмент.

Если все-таки оставлять страницу Narratives, ее нужно полностью переосмыслить:

- Показывать не список абстрактных нарративов, а clusters:
  - название;
  - описание;
  - frame;
  - supporting articles;
  - источники;
  - временная динамика;
  - главные сущности;
  - confidence и evidence coverage.
- Добавить drilldown:
  - какие статьи поддерживают нарратив;
  - какие цитаты подтверждают;
  - какие источники расходятся;
  - какие события входят.
- Дизайн привести к новому dashboard style, без старых карточек.

Но приоритет ниже, чем анализ, похожие, графы и сравнение.

## 6. LLM и качество применения

Главная проблема: LLM сейчас используется как “сделай все сразу”. Для демо это работает, но качество нестабильно.

Новая схема:

### Шаг 1. Факты

LLM возвращает:

- основные факты;
- участники;
- даты;
- места;
- прямые цитаты/фрагменты;
- что утверждается точно;
- что является оценкой.

### Шаг 2. Интерпретация

На основе фактов и evidence LLM возвращает:

- sentiment;
- stance;
- framing;
- sympathy;
- criticism;
- narrative hypothesis.

### Шаг 3. Проверка

Backend проверяет:

- JSON валиден;
- enum значения корректны;
- quote реально есть в тексте;
- confidence в диапазоне 0-1;
- entities не пустые;
- narrative не слишком общий.

### Шаг 4. Repair

Если JSON сломан:

- один retry с repair prompt;
- если снова ошибка, сохранить raw response и показать понятную ошибку.

### Модель

`qwen3:4b` подходит для локального MVP, но может подвисать на длинных prompt. Нужно:

- делать короткие prompt;
- ограничивать `num_predict`;
- показывать UI timeout;
- добавить `/health/llm`, который проверяет не только `/api/tags`, но и короткую генерацию;
- рассмотреть отдельную модель для быстрых классификаций и rerank, если локально потянет MacBook.

## 7. Практичный ближайший спринт

### Неделя 1: стабилизировать core loop

1. Упростить UI-навигацию до `Articles`, `Analysis`, `Graph`, `Compare`.
2. Скрыть Events, Source Profile, Reports, Discover Narratives из главного меню.
3. Добавить readiness badge на статьи.
4. Сделать кнопку Analyze с нормальными состояниями и ошибками Ollama.
5. Сделать empty state для похожих: “реально похожих материалов не найдено”.

### Неделя 2: качество данных

1. Добавить `jobs`.
2. Перевести analyze/embed/similar/precompute в pipeline.
3. Добавить article fingerprint.
4. Добавить Pydantic validation LLM JSON.
5. Сохранять raw LLM response.

### Неделя 3: графы

1. Разделить 3D-граф на scene/layout/interaction/ui.
2. Починить camera controls и zoom.
3. Сделать deterministic layout.
4. Добавить breadcrumb и reason для связей.
5. Оставить только полезные graph modes.

### Неделя 4: сравнение

1. Добавить pre-check перед сравнением.
2. Добавить fact table.
3. Добавить evidence quotes для различий.
4. Предрасчитывать top-3 comparisons только для ready articles.

## Итоговая продуктовая формула

MVP должен отвечать на четыре вопроса:

1. **Что в этой статье произошло и как она это подает?**
2. **Какие материалы реально похожи и почему?**
3. **Как статья связана с участниками, источником, фреймом и соседними материалами?**
4. **Чем отличается освещение одного сюжета разными источниками?**

Если функция не помогает ответить на эти вопросы, ее лучше скрыть до следующей версии.
